# ArUco 바닥 마커 측위 — ROS 2 노드 구현 계획

> 작성: 2026-07-20. 목표는 **폐루프 주행**.
> 팀원이 **카메라가 설치된 로봇 USD**를 제작 중이며, 그것이 도착하면 즉시 실행되도록 설계한다.
> 프로젝트 현재 상태는 `HANDOFF.md`, 막혔던 것은 `DEBUG_LOG.md`.

---

## 배경 — 왜 필요한가

Phase 1은 SLAM/nav2 없이 바닥 ArUco로 주행한다(HANDOFF 4-A절). **메카넘 오도메트리가
혼자서는 못 버티기 때문**이다. 자체 측정 오차 3~12%를 통로 36.2 m에 대입하면 **1.09~4.34 m**가
밀려 슬롯 폭 3.40 m를 넘는다. 3.40 m마다 마커로 보정하면 0.10~0.41 m다.

**현재 동작하는 것은 없다.** 마커 43장은 단색 `Cube`(UV 없음, ID 전부 −1)라 cv2가 검출할 수
없고, 오도메트리 퍼블리시도 없다.

## 아키텍처 — 노드 분리

**핵심 원칙: 무거운 데이터는 C++ 브리지, 가벼운 것만 파이썬.** 이미지를 파이썬 rclpy로
퍼블리시하면 Isaac 루프가 죽는다. `ROS2CameraHelper`(C++)로 내보내고, 검출은 **외부 노드**에서
한다. 그러면 Isaac은 렌더링만 하고 인식 부하를 안 진다.

```
┌─ Isaac Sim (Python 3.11) ──────────────────────────────────┐
│  aruco_sim_bringup.py  (mecanum_ros2_drive.py 확장)          │
│                                                             │
│  OmniGraph — C++ 브리지                                      │
│   ├ ROS2CameraHelper      →  /robot/front_cam/image_raw     │
│   ├ ROS2CameraInfoHelper  →  /robot/front_cam/camera_info   │
│   ├ ROS2PublishOdometry   →  /odom                          │
│   ├ ROS2PublishTransformTree → /tf                          │
│   └ ROS2SubscribeTwist    ←  /cmd_vel          (기존)        │
│                                                             │
│  내부 rclpy(3.11) — 가벼운 것만                              │
│   └ /arm_control 서비스                         (기존)        │
└─────────────────────────────────────────────────────────────┘
                        ↕ DDS (ROS_DOMAIN_ID=126)
┌─ 외부 ROS 2 Humble (3.10) — src/parkbot_aruco ─────────────┐
│  aruco_detector      /image_raw + /camera_info              │
│                       → /aruco/detections (vision_msgs)     │
│  aruco_localizer     /aruco/detections + /odom              │
│                       → /robot_pose,  /tf (map→odom)        │
│  aruco_navigator     /robot_pose + NavigateToPose 액션        │
│                       → /cmd_vel                            │
└─────────────────────────────────────────────────────────────┘
```

**이 분리가 주는 것**
- 이미지가 C++ 경로로 나가 Isaac 루프가 안 죽는다
- `camera_info`가 내부 파라미터를 자동으로 실어 나른다 — 손으로 K 행렬을 만들 필요가 없다
- 외부 노드는 시스템 python 3.10 + `cv2 4.5.4`(aruco 포함)로 돈다. `src/M0609`의
  `color_detector`가 이미 같은 구조다(`/rgb` 구독 → 결과 퍼블리시)
- `ros2 topic echo`, `rviz2`, `ros2 bag`이 그대로 먹는다. 디버깅이 훨씬 쉽다
- **`NavigateToPose`가 자연스럽게 들어간다** — HANDOFF 5절의 "Phase 2에서 nav2로 교체" 원칙이
  인터페이스 수준에서 지켜진다

## 0. 먼저 못 박을 것 — 좌표/프레임 계약

**이걸 애매하게 두면 이후 단계가 전부 부호 디버깅으로 소모된다.** 코드보다 먼저 확정한다.

| 항목 | 값 | 근거 |
|---|---|---|
| 스테이지 up | **+Y** | `build_parking_environment.py` |
| 바닥 평면 | 월드 **XZ** | `marker_layout.markers()`가 `(x, z)` 반환 |
| 로봇 배치 | 로봇 +X → 월드 +Z | `mecanum_ros2_drive.py:129-135` |
| **nav yaw** | **`ψ = atan2(fwd_x, fwd_z)`** (0 = 월드 +Z) | `mecanum_holonomic_test.py:187-188`. **새로 정의하지 말고 재사용** |
| yaw 회전축 | 월드 **+Y** (Z-up 아님) | 위에서 유도 |
| `wz` 부호 | `wz=+0.5` → yaw 증가 | `mecanum_holonomic_test.py:229`에서 이미 검증됨 |

**Y-up 함정**: nav2/REP-103 예제는 전부 Z-up 가정이다. 회전 행렬을 복붙하면 **yaw가 조용히 틀린다.**
그리고 XZ 평면에서 `atan2(x, z)`는 위에서 볼 때 시계방향 증가라 "CCW+" 직관과 반대로 보인다.
다행히 `wz`와는 부호가 맞으므로 **이 정의를 고수하고 직관을 버린다.**

marker 오브젝트 프레임(solvePnP `objectPoints`용): `X_m = 월드 +X`, `Y_m = 월드 −Z`, `Z_m = 월드 +Y`.
`Z_m`이 위(카메라 쪽)를 향해야 solvePnP가 정상 해를 낸다.

<details><summary>M1 상세 (완료)</summary>

지금 마커는 단색 쿼드라 cv2가 절대 못 본다.

**사전은 `DICT_5X5_100`** — 4×4가 아니다. 43장에 여유가 필요하고, 무엇보다 **4×4는
inter-marker Hamming 거리가 작아 오검출(엉뚱한 ID) 위험이 크다.** 바닥 마커는 반사·모션블러로
비트가 깨지기 쉬운 조건이라 이 여유가 중요하다. 픽셀 예산은 아래 계산대로 남아돈다.

ID는 kind별 블록으로 할당해 레이아웃을 고쳐도 다른 kind가 밀리지 않게 한다:
slot 0–19 / dock 20–29 / crossing 30–39 / gateway 40–49 / handoff_bay 50–59 / handoff_lane 60–79

**타일 크기와 코드 크기를 구분할 것 — 여기서 틀리면 거리가 정확히 28.6% 어긋난다**

```
TILE_CELLS = 5(코드) + 2(테두리) + 2(흰 여백) = 9
MARKER_TILE      = 0.25 m          ← USD 쿼드 크기
MARKER_CODE_SIZE = 0.25 × 7/9 = 0.194 m   ← solvePnP가 쓰는 값
```

흰 여백(quiet zone)이 없으면 검출 자체가 안 된다. 그런데 solvePnP에는 여백을 뺀 **코드 크기**를
줘야 한다. 거리 오차가 일정 비율(≈28.6%)로 나오면 바로 이 줄을 의심할 것.

**검출 해상도는 제약이 아니다. 진짜 제약은 완전 가시 구간이다.**

| | 640×480 hFOV 90°, 0.5 m 거리 |
|---|---|
| 코드 폭 | 124 px = **17.7 px/cell** (검출 하한 3~4) |

| 카메라 높이 | 바닥 발자국 L | 타일이 온전히 들어오는 구간 | 0.4 m/s·60 Hz 프레임 |
|---|---:|---:|---:|
| 0.09 m | 0.31 | **0.06 m** | ~9 (마진 없음) |
| 0.15 m | 0.52 | **0.27 m** | ~40 (충분) |

→ **타일 0.25 m는 카메라 높이 0.15 m를 전제로 한 값**이다. 팀원 실물이 0.09 m로 오면
타일을 0.15 m로 줄여야 한다. 그래서 상수 하나로 두고 **M5 스윕이 확정하게** 한다.

**만들 것**

- `marker_layout.py` — `ARUCO_DICT`, `MARKER_TILE`, `MARKER_CODE_SIZE`, `assign_ids()` 추가
- `build_marker_textures.py` (신규) — ID별 PNG(코드 픽셀이 셀 수의 정수배여야 보간 아티팩트가
  없다: 448 = 7×64) + **`marker_map.json` 내보내기**. 이 JSON이 ROS 노드가 읽는 유일한
  아티팩트다 — 노드가 `marker_layout.py`를 import하지 않아도 되고, 검사 가능한 산출물이 된다
- `build_marker_layout.py` 수정 — `Cube` → **`UsdGeom.Mesh` 쿼드 + `primvars:st`**,
  `UsdUVTexture` + `UsdPrimvarReader_float2`로 PNG 바인딩

**머티리얼에서 반드시 `specular = 0`, `roughness = 1`.** 천장 조명이 흑백 패턴에 정반사를
만들면 이진화가 깨져 검출이 통째로 실패한다. **가장 흔한 실패 모드다.**
`MARKER_Y`는 0.021 → **0.0012**로 낮춘다(기존 페인트면 0.0008 바로 위. 페인트처럼 보임).
충돌체는 붙이지 않는다 — 마커는 페인트고 로봇이 그 위를 지나간다.

**검증**: 생성한 PNG를 cv2로 되읽어 원래 ID가 나오는지(Isaac 없이 즉시) → USD의
`aruco:markerId` 재조회 → 렌더된 화면에서 검출되는지.

#### M1에서 실제로 걸린 것 두 가지

**1. cv2 aruco API가 두 인터프리터에서 완전히 다르다 — M2 전제가 깨진다**

| API | 시스템 3.10 (4.5.4) | Isaac 3.11 (4.11) |
|---|---|---|
| `generateImageMarker` / `drawMarker` | 구버전만 | 신버전만 |
| `ArucoDetector` 클래스 | ✗ | ✓ |
| `detectMarkers` 함수 | ✓ | **✗** |
| `DetectorParameters_create` | ✓ | ✗ |

OpenCV 4.7에서 aruco가 갈아엎어졌다. 겹치는 건 `getPredefinedDictionary` 하나뿐이다.
**M2의 외부 ROS 노드(3.10)는 Isaac과 같은 코드를 쓸 수 없다.**
`build_marker_textures.py` 의 `_make_detector` / `_make_marker_image` 처럼 얇은 호환
계층을 두거나, 외부 노드용으로 cv2를 4.7+ 로 올려야 한다. M2에서 결정할 것.

> 부수 교훈: 재실행 가드가 **cv2 존재**만 확인하면 4.5.4에 머물러 터진다.
> **쓸 수 있는 API가 있는지**를 봐야 한다(`_has_cv2()`).

**2. 렌더 프로덕트는 타임라인이 돌아야 프레임을 낸다**

`Camera.initialize()` 만으로는 `get_rgba()` 가 계속 빈 배열이다.
`timeline.play()` + `world.reset()` 후 `world.step(render=True)` 로 워밍업해야 한다
(라이다 데모 `run_ceiling_lidar_demo.py:130-144` 와 같은 이유).
카메라를 옮긴 직후에도 몇 스텝 돌려야 이전 프레임이 안 나온다.

### M3 — 측위 + 오도메트리 융합 (USD 불필요) ✅ **1차 완료 (2026-07-21)**

**구현 결과 요약** (`marker_localizer.py` + `m3_localize_demo.py` + `m3_drive_localize_test.py`):

- 좌표 체인 `T_world_robot = T_world_marker·inv(T_cam_marker)·inv(T_base_cam)` 작동.
  실제 Isaac 렌더 1프레임 E2E 측위 오차 **0.6cm / 0.3°** (GT 대조).
- **규약이 첫 시도에 맞았다** — 데모가 align_yaw 를 {0,90,180,270} 스윕해 GT와 대조,
  **align_yaw=0** 선택(0절 계약이 옳음). 손 추론 대신 테스트가 답을 냈다.
- `T_base_cam`은 에셋에서 직접 읽는다(손 타이핑 금지).
- A2→A7 17m 주행 융합 테스트: **6/6 마커 인식**, 인식마다 ID·월드좌표 로그.
  오도메트리만 최종 **864cm** 드리프트 vs 마커융합 **18cm**.

**아래 M3 설계에서 실측으로 갱신된 것:**

- 정기구학(FK)은 별도 함수를 만드는 대신 **모듈 IK를 `np.linalg.pinv`로 뒤집어** 유도
  → 부호 일관성 보장(`M_fk @ M_ik = I` 검증).
- 엔코더 오도메트리는 관절 '각도'가 아니라 **'각속도'**를 써야 한다(연속 회전 각도는
  wrap 되어 차분이 무의미 → 캘리브가 27.9x로 폭주).
- 스케일 캘리브는 **정상상태 + 실주행과 같은 yaw 보정 조건**에서 해야 한다(가속 전이·
  개루프 yaw 휘어짐이 스케일을 0.56x로 오판시킴 → 올바른 값 0.907x).
- **핵심 확인**: 복합운동에서 단순 메카넘 FK가 횡을 ~0.5x로 과소계산 = 메카넘 휠
  오도메트리의 실제 한계(롤러 횡슬립). 로봇은 물리적으로 차선에 붙어 있으나(GT 안정)
  바퀴 기반 자기위치 추정이 9m씩 어긋난다 → **절대위치 마커가 필수인 이유를 수치 증명.**

**남은 M3 과제**: 현재 융합은 상보필터(게인 리셋)다. 아래 설계의 3상태 KF(자세 점프
완화 + Mahalanobis 게이트 + 기각률)는 M5 관측공분산이 나온 뒤 완성한다.

### M3 (원 설계) — 측위 + 오도메트리 융합 (USD 불필요)

좌표 변환 체인:
```
T_world_robot = T_world_marker · inv(T_cam_marker) · inv(T_robot_cam)
                      ↑                 ↑                   ↑
              marker_map.json       solvePnP            /tf (Isaac이 발행)
```
마커가 전부 바닥에 누워 있어 자유도는 실질 (x, z, yaw) 3개다. **0절 계약을 그대로 쓴다.**

**먼저 진짜 오도메트리를 만든다.** 현재 "오도메트리"라 부를 만한 것은 **명령한 `cmd_vel`을
적분**한 것뿐이라 슬립을 전혀 못 본다. `mecanum_drive.py`에 정기구학을 추가한다:

```python
def cmd_vel_from_wheel_velocities(omegas):   # wheel_velocities_from_cmd_vel 의 역
```
실제 휠 각속도(`get_joint_velocities()`)를 역변환하면 진짜 엔코더 오도메트리가 된다.
슬립 자체는 못 잡아도 **명령/실현 차이는 잡히므로** 이것만으로 드리프트가 눈에 띄게 줄 가능성이
높다. 기존 IK 함수 바로 옆에 두고 왕복 일치를 유닛테스트로 건다.

**융합은 단순 리셋이 아니라 3상태 KF로 한다.**

- *단순 리셋을 기각하는 이유*: 마커를 만날 때 자세가 0.2 m 점프한다. 폐루프 컨트롤러가
  이걸 오차로 읽으면 로봇이 **움찔한다**. 차량을 들고 있을 때 그 움찔은 옆칸 차량을 미는
  문제(HANDOFF 7절, 실측 0.38~0.43 m)를 악화시킨다.
- *풀 EKF를 기각하는 이유*: 메카넘 롤러 슬립은 비선형이고 `YAW_SCALE=1.12`도 한 동작점에서만
  맞춘 값이다. 근거 없는 걸 모델링하면 튜닝 불가능한 파라미터만 늘어난다.
- *채택*: 상태 `[X, Z, ψ]`, 공분산 3×3, **이동거리 비례 프로세스 노이즈**. numpy만으로 60줄.
  `SIG_POS ≈ 0.06`(실측 3~12%의 중앙값), `SIG_YAW ≈ 0.12`(YAW_SCALE 보정량) —
  **둘 다 이미 측정된 값이라 새로 튜닝할 필요가 없다.** 관측 공분산 `R`은 M5가 낸 값을 쓴다.
  Mahalanobis 게이트로 이상치 기각, `last_fix_distance_m`을 노출해 컨트롤러가 쓰게 한다.

**검증**: 합성 시퀀스(알려진 궤적 + 인위적 드리프트)로 ①순수 추측항법 ②단순 리셋 ③KF를
나란히 비교. **KF가 오차는 리셋과 비슷하되 자세 점프가 작아야** 통과. 리포트에 **기각률**을
반드시 출력하고 5%를 넘으면 경고(게이트가 빡세면 정상 픽스를 다 버리고 추측항법으로 전락한다).

### M4 — 팀원 USD 연결 (여기서 처음 USD 필요)

- 인수 체크리스트 확인 → 어긋난 항목 협의
- `robot_camera.py` (신규) — USD에서 **카메라 프림을 탐색**하고 역할(front/left/right)에
  매핑. 만드는 게 아니라 **찾는다**. 프림 경로가 바뀌어도 이 파일만 고치면 되게
- `aruco_sim_bringup.py` (신규, `mecanum_ros2_drive.py` 확장) — OmniGraph에
  `ROS2CameraHelper` + `ROS2CameraInfoHelper` + `ROS2PublishOdometry` +
  `ROS2PublishTransformTree` 추가. `build_cmd_vel_graph()`(161-186)와 같은
  `og.Controller.edit` 패턴이며, 기존 `OnTick.outputs:tick`에서 분기하면 된다

**검증 — 먼저 1-마커 캘리브레이션부터.** 마커 43장을 한꺼번에 돌리기 전에, **마커 1장 위에
알려진 자세로 로봇을 세우고** 복원된 `T_world_marker`가 저작한 값과 같은지 확인한다.
90°/180° 어긋나면 st 매핑이나 마커 yaw를 한 번 전역으로 뒤집으면 끝난다.

> **마커 방향을 머리로 맞히려 하지 말 것.** 우수계·이미지 up 방향·와인딩이 얽혀 첫 시도에
> 맞을 확률이 낮다. 이 1-마커 테스트를 만들고 **테스트가 답을 알려주게** 한다. 한 번 맞추면
> 43장 전부 같은 방향이라 끝난다. 이걸 건너뛰고 M5로 직행하면 실패 원인이 좌표 부호인지
> 텍스처인지 카메라 변환인지 구분이 안 되어 시간을 크게 잃는다.

그다음 `ros2 topic hz /robot/front_cam/image_raw`로 프레임이 나오는지,
`ros2 topic echo /aruco/detections`로 실제 씬 마커가 잡히는지.

### M5 — 정확도 검증 (관문)

`aruco_accuracy_test.py` — 로봇을 여러 위치·자세에 놓고 추정치를 **Isaac ground truth**
(`robot.get_world_poses()`)와 비교. 헤드리스 + JSON 리포트.

| 지표 | 용도 | 통과 기준(제안) |
|---|---|---|
| **오검출률** (검출 ID ≠ 최근접 마커 ID) | 치명적 — 로봇을 엉뚱한 곳으로 순간이동시킨다 | **정확히 0** |
| `pos_err_m` mean/median/**p95**/max | M3 필터의 `R` | p95 ≤ 0.03 m |
| `yaw_err_deg` 동일 통계 | M3 필터의 `R` | p95 ≤ 1.5° |
| **오차 공분산 3×3** | ← M3 KF의 `R`에 그대로 투입 | — |
| `reproj_err_px` 분포 | 게이트 임계 결정 | < 1.5 px |
| **마커당 가시 주행거리** | M6 커버리지 | ≥ 0.15 m |
| **차선 커버리지** (검출 가능한 호길이 비율) | 추측항법 구간 길이 | — |
| 검출 지연 | 폐루프 안정성 | — |

**스윕 기능을 넣는다**: `--sweep marker_tile=0.15,0.20,0.25` / `--sweep cam_height=0.09,0.12,0.15`
→ 조합별 리포트. **이게 "마커 크기"와 "카메라 높이"라는 두 미결 항목을 논쟁이 아니라
데이터로 닫는 장치다.** 스윕이 마커 크기를 바꾸면 M1 빌더를 서브프로세스로 재호출해야 한다.

정적 격자 모드(타임라인 정지 + `set_world_poses`로 순간이동)가 빠르고 결정적이다.
**단, 순간이동 직후 렌더하면 이전 프레임이 나온다.** 최소 2~3회 `world.step(render=True)`가
필요하고, 안 하면 오차가 "격자 한 칸만큼" 일정하게 나오는 특징적 패턴이 보인다.

> **폴백이 실패를 통과시키지 않게 할 것.** 운반 데모에서 유도 폴백이 리프트 0.4 mm를
> PASS로 만든 전례가 있다(`DEBUG_LOG.md` 2026-07-20 밤). 검출 실패 시 이전 추정치를
> 재사용한다면 **폴백 사용 여부 자체를 리포트와 판정에 넣는다.**

**M5가 관문이다.** 정확도가 안 나오면 M6은 의미가 없다.

### M5.5 — 마커 그래프 경로계획 (USD 불필요)

목표 마커 ID를 받아 **거기까지 가는 마커 시퀀스**를 만든다. M6 컨트롤러는 "다음 마커까지"만
담당하고, 경로가 그 시퀀스를 공급한다.

**격자 A*도 코스트맵도 필요 없다 — 마커 배치 자체가 그래프다.** 노드 44개짜리 Dijkstra는
1 ms도 안 걸린다. Phase 1에 SLAM이 없는 것과 일관된다.

```
"7번으로 가라"  →  marker_map.json 조회  →  그래프 최단경로  →  마커 시퀀스
                                                                    ↓
                                                     구간마다 M6가 주행 + 마커에서 자세 리셋
```

#### 엣지 규칙 — 거리 제한이 물리에서 나온다

경로를 마커를 따라 잡는 이유는 **장애물 회피가 아니다.** 통로 z ∈ [−4.5, 4.5]는 9 m 폭의
빈 바닥이라 대각선으로 질러가도 안 부딪힌다. 진짜 이유는 **다음 마커를 다시 찾기 위해서**다.
무보정으로 멀리 가면 옆으로 밀려 마커가 시야 밖으로 나간다.

```
MAX_LEG = (마커 가시구간 ÷ 2) ÷ 횡방향 드리프트율
```

| 카메라 높이 | 가시구간 | **MAX_LEG** | 차선 간격 3.40 m 대비 |
|---|---:|---:|---|
| **0.15 m** | 0.27 m | **5.00 m** | 여유 있음 |
| **0.09 m** | 0.06 m | **1.11 m** | **차선을 따라가는 것조차 실패** |

횡방향 드리프트율 2.7%는 실측값이다(strafe 4.37 m 중 0.12 m).

> **이것이 카메라 높이가 취향 문제가 아닌 이유다.** 0.09 m면 마커를 1 m 간격으로 100장 넘게
> 깔거나 카메라를 올려야 한다. 인수 체크리스트 7번 참고.

#### 실측된 개선

`MAX_LEG = 5.0 m`(카메라 0.15 m 전제)로 "5 m 이내면 잇는다" 규칙을 쓴 결과:

| 경로 | 직선 | 같은-x 규칙 | **MAX_LEG 규칙** | 우회율 |
|---|---:|---:|---:|---:|
| A5→B3 | 8.44 | 18.60 | **11.80** | 1.4배 |
| A3→B6 | 11.36 | 22.00 | **15.20** | 1.3배 |
| A1→B7 | 21.00 | 25.40 | **22.04** | 1.0배 |
| A1→A8 | 23.80 | 23.80 | **23.80** | 1.0배 |

엣지 28 → 53개, **평균 우회율 2.2배 → 1.19배**. `A5→B3`이 `A5→A4→A3→B3`으로
대각선 횡단을 쓴다.

#### 이 분석이 배치 결함을 잡아냈다

경로탐색을 붙이자마자 두 가지가 드러났다. **경로탐색 결과가 마커 배치의 검증 도구가 된다.**

- 횡단 마커가 3곳(x = −15.3, 0, +15.3)뿐이라 `A5→B3`이 32.2 m로 돌아갔다 →
  A2/B2(−8.5)와 A7/B7(+8.5)에 추가해 18.6 m로 단축
- x=0.0 마커는 **어느 슬롯 열과도 x가 안 맞아 고립 노드**였다(열은 ±1.7, ±5.1, …) → 제거.
  마커 43 → 44장

#### 적재/무적재 그래프가 달라야 한다

차를 들면 점유 폭이 4.8~5.8 m로 커지고 **회전이 금지된다**(HANDOFF 5절). 옆칸 차량을
밀면 안 된다(실측 0.38~0.43 m 밀림, HANDOFF 7절). 무적재는 자유롭게, 적재는 차선을
벗어나지 않게 — **두 벌의 엣지 집합**을 둔다.

**만들 것**: `marker_graph.py` (순수 파이썬, 의존성 없음). `marker_map.json`을 읽어
그래프 구성 + Dijkstra. 현재는 일회성 분석 스크립트로만 존재한다.

**검증**: 슬롯 쌍 전수(16×16)에 대해 ①경로 존재 ②모든 구간 ≤ MAX_LEG ③우회율 통계.
고립 노드 목록 출력 — 인계장 18장은 동선 미해결이라 당분간 고립이 정상이다.

### M6 — 폐루프 주행

`aruco_navigator.py` — `/robot_pose`를 받아 목표까지 `/cmd_vel` 생성. 홀로노믹 P 제어.

이 프로젝트 특유의 요구 셋:

1. **`hold_yaw` 모드** — 적재 중에는 yaw 고정, vx/vy만으로 이동(HANDOFF 5절 "차량을 회전시키지
   않는 평행이동"). 메카넘을 쓰는 이유 자체다. 전역 설정이 아니라 **`NavGoal`의 필드**로 둔다
   (무적재 로봇은 자유 회전 가능하므로).
2. **최종 정렬은 융합 자세가 아니라 마커 직접 측정으로** — 목표 마커가 시야에 들어오면
   servo 모드로 전환해 `T_cam_marker`로 직접 제어. 누적 오차가 0이라 정지 오차가 훨씬 작다.
3. **픽스 신선도 기반 감속** — `last_fix_distance_m`이 크면 최대속도를 낮춘다.
   3.40 m마다만 보이는 조건에서 신뢰도와 속도를 연동한다.

**적분항(I)을 쓰지 말 것** — 마커 부재 구간에서 와인드업한다. 대신 도착 판정 + 재수렴 루프.

**`NavigateToPose` 액션 서버로 감싼다.** 미션 코드는 이 인터페이스만 알고,
Phase 2에서 nav2로 바꿀 때 백엔드 교체로 끝난다:

```
NavigateToPose  ← 미션 코드는 이것만 안다
   ├ ArucoNavBackend   (Phase 1)
   └ Nav2Backend       (Phase 2)
```

**리스크**: `YAW_SCALE=1.12`는 wz≈0.5 rad/s 동작점에서만 맞춘 경험값이라, 폐루프가
넓은 wz 범위를 쓰면 여기가 먼저 틀어진다. 회전 구간 오차가 유독 크면 이걸 의심할 것.

## 순서와 관문

```
USD 없이 진행 가능 ──────────────────────┐
 M1  마커 텍스처·ID                        │
 M2  검출 노드        (합성 이미지로 검증)   │
 M3  측위·오도메트리   (합성 시퀀스로 검증)   │
 M5.5 마커 그래프 경로계획 (전수 경로 검증)   │
─────────────────────────────────────────┘
          ↓  팀원 USD 도착
 M4  USD 연결 (카메라 탐색 + OmniGraph 배선)
          ↓
 M5  정확도 검증  ← 관문. 여기서 못 넘으면 M6 무의미
          ↓        (MAX_LEG 실측값이 여기서 확정 → M5.5 재검증)
 M6  폐루프 주행 + NavigateToPose
```

M5.5는 언제 해도 되지만 **M5가 낸 실측 드리프트율로 `MAX_LEG`를 다시 계산**해야 한다.
지금 쓰는 2.7%는 strafe 시험 한 건에서 나온 값이다.

M1~M3를 미리 끝내두면, USD가 도착했을 때 **M4의 배선 작업만 하면 바로 돌아간다.**

## 범위에서 제외

- **인계장 동선** — 베이 열 사이 0.70 m라 차를 싣고 지나갈 길이 없다(HANDOFF 4-A절).
  팀원과 조율 전까지 마커도 잠정
- **좌우 카메라** — USD에 있으면 M4에서 `robot_camera.py` 매핑에 추가. 없으면 front만으로 시작
- **차종 일반화** — 물리 쪽 별건(HANDOFF 7절 최우선 리스크)
