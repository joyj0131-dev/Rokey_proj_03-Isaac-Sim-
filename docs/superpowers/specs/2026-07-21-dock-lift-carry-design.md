# 2로봇 순차 도킹 + 파지 + 리프트 + 운반 (Plan 3) — 설계 (ROS2 촉발)

> 작성: 2026-07-21. 관련: `two-robot-fleet-design.md` v3(E2E 미션 3~5단계),
> `two_robot_carry_demo.py`(검증된 grip+lift+carry 물리), Plan 1(`/arm_control`
> ROS2 서비스 패턴), Plan 2(외부 ROS2 미션 오케스트레이션 패턴).

## 목표 (사용자 정의)

> "프로그램을 켜둔 뒤, **순수 ROS2에서 요청 한 번**을 보내면 — 인계장에 있는
> 차 밑으로 두 로봇이 **순서대로** 들어가서, **바퀴감지 없이 하드코딩된 정지
> 위치**에서 바퀴를 잡고, 들어올려서 이동한다."

즉 `ros2 service call /dock_lift ...` 한 번 → 진입→파지→리프트→운반 전 과정.

## 범위

**포함**: Isaac 러너가 팔 파지/리프트를 ROS2 서비스로 노출 + 외부 ROS2
오케스트레이터가 요청 하나로 2로봇 순차 도킹→파지→리프트→운반을 구동.
헤드리스 검증 + GUI 시연.

**제외**:
- 바퀴/뎁스 센서 감지 (사용자 명시적 제외 — 정지 위치는 차량 축 좌표에서 하드코딩)
- Plan 2의 도크→인계장 ArUco 주행과의 체이닝 (이미 동작. 통합은 후속.
  단 인터페이스는 맞춰 둔다 — 둘 다 외부 ROS2 요청 기반이라 자연히 이어짐)
- 차종 일반화 (A5_Coupe 고정 — 2로봇 리프트 검증 차종)

## 아키텍처 — Isaac은 브리지·서비스, 제어는 외부 ROS2 (프로젝트 규약 준수)

```
┌─ Isaac 듀얼 러너 (run_dual_robot_ros2_field.py 확장) ── 시작 후 대기 ─┐
│  씬: 인계장 + A5_Coupe(베이 배치, 콜라이더 수정, 파지 마찰재) + 로봇 2대 │
│  C++ 브리지: /robot_N/cmd_vel 구독, /robot_N/odom·wheel_twist 발행(기존) │
│  내부 rclpy(3.11) 서비스(신규):                                        │
│    /robot_N/arm_control (std_srvs/SetBool) — true=파지·리프트/false=해제 │
│    · 팔 스윙 조인트를 ARM_TARGETS(검증각)로 램프 구동, 다 전개 후 응답    │
│    · Plan 1의 /arm_control 패턴 그대로(mecanum_ros2_drive.py 검증됨)     │
└──────────────────────────────────────────────────────────────────────┘
                    ↕ DDS (ROS_DOMAIN_ID=126, 화이트리스트 off)
┌─ 외부 ROS2 오케스트레이터 (dock_lift_mission_node, 신규) ──────────────┐
│  서비스 제공: /dock_lift (std_srvs/Trigger 또는 자체 srv) — 이게 "요청" │
│  요청 수신 시 상태기계:                                                 │
│   1. INGRESS_FAR : robot_far 에 cmd_vel(vx>0) → 먼 축 하드코딩 위치 정지 │
│   2. INGRESS_NEAR: robot_far 진입 확인 후 robot_near 진입·정지 (순차)    │
│   3. GRIP        : 두 로봇 /arm_control(true) 병렬 호출 → 파지·리프트     │
│   4. CARRY       : 편대 cmd_vel 로 차 든 채 CARRY_DISTANCE 이동           │
│   5. DONE        : 응답(success + 리프트/추종 지표), 리포트 JSON          │
│  측위: /robot_N/odom (GT, Plan 1·2 규약). 정지 판정 = 목표 위치 도달.    │
└──────────────────────────────────────────────────────────────────────┘
```

## 재사용 (위험 최소화)

`two_robot_carry_demo.py`가 A5_Coupe로 검증한 **물리**를 그대로 가져온다 —
단 제어 주체만 Isaac 내부 → ROS2로 바뀐다:
- **씬·차량·축 기하**: A5_Coupe 배치, 실제 휠 좌표에서 축 위치 산출(하드코딩 좌표 근거)
- **파지**: `ARM_TARGETS`(팔 전개각) + 파지 마찰재(static 1.2) — Isaac 서비스가 구동
- **운반**: `CARRY_SPEED`·facing 부호 — 외부 오케스트레이터가 cmd_vel 로 구동
- **측정**: `car_lift_m`, `carry_follow_ratio`, `grip_jitter_ratio`

`/arm_control` 서비스 골격은 Plan 1의 `mecanum_ros2_drive.py`에서 이미 검증됨
(내부 rclpy, SetBool, "다 열린 뒤 응답"). 그것을 듀얼 러너에 로봇별로 편입.

## 진입 정지 위치 (하드코딩)

차량 축 좌표에서 계산. 로봇은 자기 grip 목표(x/z/yaw)로 cmd_vel P제어 직진,
목표 도달(위치 오차 < 허용) 시 정지. 센서 감지 없음. 순차 이유: 좁은 하부에
동시 진입 시 충돌 — 먼 축 로봇이 자리잡은 뒤 가까운 축 진입.

## 콜라이더 수정

A5_Coupe 바퀴 콜라이더가 시각 대비 1.61배(타이어 안쪽 간격 콜라이더 0.99m vs
시각 1.18m, 로봇 폭 ~0.95m — 4cm 여유뿐). 러너의 씬 구성 단계에서 각 바퀴
`Collision.height`를 시각 폭으로 override(원본 `fab_vehicles.usd` 비파괴,
구동계 raycast 서스펜션 무관). 수정 후 1.18m → 로봇 0.95m가 0.23m 여유 진입.

## 좌표/제약

- 진입은 **직진만** (제자리 회전 비신뢰 — Plan 2 실측). 로봇은 진입 방향을 향한
  채 스폰/도착.
- 로봇 에셋은 낮춘 바퀴 변형(`_lowered.usd`, 지상고 +3cm) 사용.
- cmd_vel yaw 부호·화이트리스트 off·좌표 규약은 Plan 1·2 확정값 그대로.
- GPU PhysX, 원본 USD 비파괴, 좀비 프로세스 정리.

## 에러 처리 (실패는 정지 방향)

| 상황 | 동작 |
|---|---|
| 진입 중 목표 미도달(타임아웃) | 해당 로봇 cmd_vel=0, 오케스트레이터 실패 응답 + 사유 |
| arm_control 무응답/타임아웃 | 파지 중단, 실패 응답 |
| 리프트 판정 미달(car_lift_m 낮음) | CARRY 진입 안 함, 실패 응답(어느 단계 실패인지 명시) |
| odom 끊김 | 전체 정지 |

## 검증 (도착 좌표만 보지 않는다 — Plan 2 교훈)

1. **ROS2 촉발 실동작**: 외부 터미널 `ros2 service call /dock_lift` 한 번 →
   전 과정 자동 진행, success 응답. (핵심 성공 기준)
2. **리프트**: `car_lift_m ≥ 0.025 m`
3. **운반**: `carry_follow_ratio ≈ 1.0`, 실제 구동 거리 ≥ 목표
4. **모션품질**(`motion_quality_probe.py` 재사용): 진입·운반 중 두 로봇 발레·점프·
   이탈 없음. **통과 못 하면 PASS 아님.**
5. **옆칸 차량 무변위**: 대상 외 차량 최대 변위 낮음.

## 테스트 계획

1. 단위(Isaac 불필요): 오케스트레이터 상태기계 전이 로직(진입 완료 판정, 순차
   게이트, 실패 전이)을 순수 함수로 분리해 단위 테스트.
2. 헤드리스: 러너 + 오케스트레이터 기동 → `ros2 service call /dock_lift` →
   위 5개 판정 콘솔 + JSON.
3. GUI 시연 — 순차 진입·파지·리프트·운반 육안 확인.
4. 실패 시 리포트 지표로 단계 격리(진입 vs 리프트 vs 운반).

## 리스크

- A5_Coupe 리프트는 carry demo에서 **순간이동 진입 + Isaac 내부 제어** 기준 검증.
  **실제 cmd_vel 구동 진입 + ROS2 서비스 파지로 바꿨을 때** 파지 정렬이 어긋날 수
  있음 — 진입 정지 위치 정밀도가 관건. 하드코딩 좌표 + P제어 + 필요시 튜닝.
- 낮춘 바퀴(+3cm)가 carry demo 검증 조건과 다름 — 리프트 재현 재확인.
- 서비스 파지의 팔 램프 속도가 물리 안정성에 영향 — 검증된 램프 스텝 수 재사용.
- 순차 진입 타이밍(먼 축 로봇 진입 완료 판정 기준) — 실측 조정.
