# 파지 차량 가상중심(virtual structure) 협조제어 설계

> **상태:** 설계 노트(아이디어 회의 결과). **구현 보류** — 팀원이 Plan 3 기반 ROS2 구조를
> 완성해 인계받은 뒤, "차를 잡았을 때의 제어(운반 단계)"를 이 방식으로 교체한다.
> 작성 2026-07-22.

## 목표 (한 줄)

두 로봇이 차를 잡은 상태에서, 두 로봇+차를 **하나의 가상 강체**로 보고 그 중심을
제어함으로써 — 지금은 안 되는 **회전**을 포함해 임의의 평면운동(병진+회전)을 구현한다.

## 배경 / 현 상태

현재 인계장 데모(`dock_lift_handoff_mission.py`)의 운반(`_omni_carry`)은:
- 두 로봇이 **같은 자세(-z 향함)** 라, **동일한 cmd_vel(vx, vy, ω=0)** 를 줘서 편대 병진만 함.
- 회전을 못 넣는 이유: 두 로봇에 같은 ω를 주면 **각자 자기 중심으로** 돌아 차를 비틈.
- 결과: 앞/뒤/옆 병진은 되지만 **차 방향(heading)을 못 바꿈** → 슬롯에 각 맞춰 넣기 불가.

## 근거가 된 실측 (중요)

메카넘 "30% 슬립"은 **실제 슬립이 아니라 real-time factor 아티팩트**였음(2026-07-22 실측,
`friction_experiment.py` / `rtf_check.py`):
- 깨끗한 씬 sim시간 측정: 지령의 **101%** (마찰 1.0/3.0 동일 → 마찰 무관, 슬립 없음).
- 무거운 주차장 씬: sim시간 101%, 벽시계 37%, **RTF=0.37**.

→ **로봇이 지령을 ~1:1로 정확히 따른다**는 뜻. 따라서 가상강체 분해로 나온 각 로봇 지령을
로봇들이 충실히 추종하므로, 두 로봇이 어긋나 그립을 비트는 문제는 거의 없다. (폐루프는
슬립 보정용이 아니라 초기오차·잔차 정리 + (c) 확장용으로 유지.)

## 설계

### 제어 프레임 & 피드백 (로봇 odom만 사용)

- **가상 중심 C** = 두 로봇 위치의 중점: `C = (p_rear + p_front) / 2`
- **회전량(heading) θ** = 두 로봇을 잇는 baseline 각도: `θ = atan2(p_front − p_rear)`
- 둘 다 **두 로봇 odom만으로 계산** → `/vehicle/pose`(차 pose) 불필요. 실차 이식에 유리.

```
        p_front
          o──────── baseline (각도 θ)
         /
        C  ← 가상 중심(중점)
       /
      o
   p_rear

  회전: 각 로봇이 C 기준 반지름 r_i 의 접선 방향으로 옆이동 + 자전 Ω
        (사다리를 둘이 들고 도는 것과 동일)
```

### 컨트롤러 (가상강체 3-DOF, 매 틱)

월드는 XZ 지면 평면 + Y up, 회전 Ω는 Y축 기준. 위치·속도는 (x, z) 성분.

```
입력: 목표 회전량 Δθ
초기: C0 = C,  θ* = θ + Δθ

매 틱:
  C = (p_rear + p_front)/2                       # (Cx, Cz)
  θ = atan2(p_front − p_rear)                    # baseline 각도
  V = (Vx, Vz) = Kp · (C0 − C)                   # 중심 고정(드리프트 억제)
  Ω = clamp(Kθ · wrap(θ* − θ), Ω_max)            # Y축 회전

  for i in {rear, front}:                        # 강체 운동 → 각 로봇 지령 분배
      r_i = p_i − C = (r_ix, r_iz)
      # 접선속도 = Ω × r_i (Y축 회전). 성분 부호(±)는 odom 규약에 맞춰 실증 확인:
      #   V_C + Ω ẑ×r 형태이며, ground-frame(X,Z)이 회전축과 이루는 손잡이에 따라
      #   부호가 갈린다. 앞서 strafe(vy) 부호가 뒤집혀 있던 전례가 있으므로
      #   (dock_lift_handoff_mission 참고) 반드시 소량 회전으로 방향 검증 후 확정할 것.
      v_i = V + Ω × r_i                           # ≈ (Vx ± Ω·r_iz,  Vz ∓ Ω·r_ix)
      body = world→body(v_i, yaw_i)               # 로봇 i 자세로 회전변환(부호 검증 필요)
      publish cmd_vel_i = (body.x, body.y, angular = ±Ω)   # 자전 부호도 검증

종료: |wrap(θ* − θ)| < 허용치
```

> **부호 주의:** θ 측정(atan2)·접선속도(Ω×r)·world→body 변환·자전(±Ω)의 부호가 서로
> 일관돼야 한다. 어긋나면 양의 되먹임으로 회전이 발산한다. 실제 odom/메카넘 규약에
> 이미 부호 반전 사례가 있었으므로(strafe), 구현 시 **작은 Δθ로 방향부터 실증**하고 고정.

### 세 단계는 같은 컨트롤러, 입력만 다름

- **(a) 제자리 회전 [1차 목표]**: 중심고정(Vx,Vy=Kp(C0−C)) + 목표각 Δθ.
- **(b) 곡선 운반**: 중심에 목표속도(Vx,Vy≠0)와 Ω를 동시에 → 호(arc).
- **(c) 목표 pose 주행**: 바깥에 pose P루프 `(Vx,Vy,Ω)=Kp·(pose* − pose)` → 사실상
  가상 물체 NavigateToPose(위치+방향). 슬롯에 각까지 맞춰 진입.

## 제약 & 검증 포인트

- **Ω 상한**: 각 로봇이 중심에서 |r_i|≈1.8m(휠베이스 3.59m의 절반). 접선속도=Ω·|r_i|이
  로봇 최대속도를 안 넘게 `Ω_max ≈ v_max / 1.8` (대략 0.3 rad/s → 90°가 sim ~5s).
- **유일한 물리적 관문 — 그립 전단**: 회전 시 바퀴가 팔 안에서 미끄러지지 않고 차가 로봇과
  함께 도는가. 병진 운반은 검증됨(그립 마찰 static 1.2). 회전은 비트는 힘이라 별도 검증 필요.
  실패 시 대응: 그립 마찰↑ 또는 팔 각도(파지 형상) 조정.
- **자세 일관성**: 파지 후 두 로봇+차는 강체라 각 로봇을 독립 제어 불가. 지령은 반드시
  "C 기준 강체 회전"과 일관돼야 함(위 분배식이 이를 보장). 불일치 지령은 그립을 비틈.

## 통합 계획

- **적용 위치**: 현재 `_omni_carry`(파지 후 운반 제어)를 이 가상중심 컨트롤러로 교체.
  파지 전 자유이동(각 로봇 웨이포인트 P제어)은 그대로 둔다 — 강체가 아니므로 별개.
- **선행 조건**: 팀원이 Plan 3 코드를 ROS2 구조(액션/노드)로 완성 → 인계받은 뒤,
  로봇 제어 계층에서 "파지 상태 제어"만 이 방식으로 스왑.
- **입출력 계약**: 입력 = 두 로봇 odom(선택적으로 차 pose). 출력 = 로봇별 cmd_vel.
  회전 목표 Δθ(또는 (c)의 목표 pose)를 상위(오케스트레이터/액션 goal)에서 받음.

## 범위 밖 (지금 안 함)

- 팀원 ROS2 구조 인계 전 구현 착수 안 함.
- 파지 전 내비게이션·마커 측위·관제(존락 등)는 이 노트 범위 밖.
- 이 노트는 **파지된 차의 운반 제어 방식**만 다룬다.
